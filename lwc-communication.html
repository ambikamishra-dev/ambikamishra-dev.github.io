@api down, events up, LMS sideways: how LWC components should talk

Ambika Mishra · 2026 · 5 min read

Most LWC bugs I have debugged were not JavaScript bugs. They were communication bugs: a child mutating its parent's data, an event that never bubbled where anyone expected, two unrelated components polling the server because neither knew the other existed. The framework has exactly three sanctioned channels, one per direction, and components stay maintainable precisely as long as everyone respects the directions.

Down: @api. A parent passes data to a child through public properties, and calls behavior through public methods. The child exposes the interface; the parent drives it.
Up: events. A child never reaches into its parent. It announces what happened with a CustomEvent and lets whoever is listening decide what to do about it.
Sideways: Lightning Message Service. Components with no ancestor relationship, anywhere on the page, publish and subscribe over a message channel. It even crosses into Aura and Visualforce, which makes it the bridge during long migrations.

Down, and the trap inside it

The mechanics are one decorator: the child marks a property @api recordId, the parent binds it in markup. The trap is subtler: @api data is owned by the parent, and the child must treat it as read-only. Objects and arrays arrive by reference, and a child that mutates them has silently created shared state with its parent. When something needs to change, the child copies first or, better, asks upward through an event. One-way data flow is not a style preference; it is what makes a component tree debuggable, because every value has exactly one owner.

Up, and the naming that saves you later

// child: announce, don't command
this.dispatchEvent(new CustomEvent('caseselect', {
    detail: { caseId: this.selectedId }
}));
<!-- parent: listen and decide -->
<c-case-list oncaseselect={handleCaseSelect}></c-case-list>

Two habits carry the weight here. Name events after what happened, not what should happen next: caseselect, not opendetailpanel. The child reporting news stays reusable anywhere; the child issuing commands is welded to one parent's intentions. And keep detail payloads primitive where possible; shipping whole mutable objects upward reopens the shared-state door you just closed.

Sideways, and when it is actually justified

LMS is the channel people reach for too early. The test I apply: if the components share an ancestor you control, communicate through the ancestor. Parent-mediated communication keeps the data flow visible in one file. LMS earns its place when there is genuinely no shared ancestor: a utility bar component talking to a record page, two regions of an app page, or an LWC coordinating with legacy Aura during a migration. It is a page-wide broadcast, and broadcasts used casually become the global variables of the UI layer.

every value should have exactly one owner. every event should be news, not orders.

The review checklist

  1. Does any child write to an @api property or mutate an @api object? That is the shared-state bug waiting for a bad day.
  2. Are events named as facts (selected, changed, submitted) rather than instructions?
  3. Could this LMS channel be a parent handler instead? If yes, it should be.
  4. Does anything sideways-communicate through the DOM, window objects, or pub-sub hacks predating LMS? Migrate it; the sanctioned channel exists now.

The opinion part

I think component communication is where LWC quietly enforces good architecture. The framework makes the right thing (one-way data, announced events) the easy thing, and the wrong thing (reaching across the tree) genuinely awkward. When a component tree feels painful, the pain is usually a message: some data has two owners, or some event is secretly a command. Fix the ownership and the tree fixes itself.
