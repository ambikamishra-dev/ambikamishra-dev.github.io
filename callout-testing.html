Testing Salesforce callouts beyond the happy path

Ambika Mishra · 2026 · 6 min read

Most callout test suites I have reviewed share the same shape: one mock, one 200 response, one assertion that the data parsed. The suite is green, coverage looks healthy, and the first time the external system returns a 500 at two in the morning, nobody knows what the code actually does, because nobody ever asked it.

Integrations fail. The vendor API times out, the middleware layer has a deployment window, the other Salesforce org in a multi-org sync throws a limit error. Whether the far side is MuleSoft, a fulfillment vendor, or another org, the shape of the problem is identical, and so is the discipline: the failure paths deserve tests before the happy path does, because the happy path is the one you will notice breaking anyway.

Why Salesforce forces mocking, and why that is a gift

Apex tests are not allowed to make real callouts. The platform blocks them so tests stay deterministic and never depend on someone else's uptime. The escape hatch is HttpCalloutMock: you hand the test a fake HTTP layer, and your class runs against responses you control completely.

That constraint is the gift. Since you must fabricate the response anyway, fabricating a 500 costs exactly as much as fabricating a 200. There is no excuse left for testing only sunshine.

The service under test

The example is from my CaseDesk project: an order-status service calling an external fulfillment API through a Named Credential. The part that matters is the status-code branching:

public with sharing class OrderStatusService {

    public static String getStatus(String orderNumber) {
        HttpRequest req = new HttpRequest();
        req.setEndpoint('callout:Fulfillment_API/orders/' + orderNumber);
        req.setMethod('GET');
        req.setTimeout(10000);

        HttpResponse res = new Http().send(req);

        if (res.getStatusCode() == 200) {
            OrderResponse parsed = (OrderResponse)
                JSON.deserialize(res.getBody(), OrderResponse.class);
            return parsed.status;
        } else if (res.getStatusCode() >= 500) {
            IntegrationLogger.log(req, res);   // transient: log, retry later
            return 'UNAVAILABLE';
        } else {
            IntegrationLogger.log(req, res);   // 4xx: our request is wrong
            return 'INVALID_ORDER';            // retrying will not help
        }
    }
}

The 4xx versus 5xx split is the heart of the design. A 5xx means the other side is having a bad day, so the caller may retry with backoff. A 4xx means the request itself is wrong, and retrying a wrong request just generates load and noise. Code that treats every non-200 the same way has not made a decision; it has avoided one.

one mock per behavior, not one mock per suite

The mocks: one per behavior

@isTest
global class FulfillmentMock200 implements HttpCalloutMock {
    global HttpResponse respond(HttpRequest req) {
        HttpResponse res = new HttpResponse();
        res.setStatusCode(200);
        res.setBody('{"status":"SHIPPED","eta":"2026-07-12"}');
        return res;
    }
}

@isTest
global class FulfillmentMock500 implements HttpCalloutMock {
    global HttpResponse respond(HttpRequest req) {
        HttpResponse res = new HttpResponse();
        res.setStatusCode(500);
        res.setBody('{"error":"internal"}');
        return res;
    }
}

And the tests that use them:

@isTest
static void statusParsesOnSuccess() {
    Test.setMock(HttpCalloutMock.class, new FulfillmentMock200());
    Test.startTest();
    String result = OrderStatusService.getStatus('4521');
    Test.stopTest();
    System.assertEquals('SHIPPED', result);
}

@isTest
static void serverErrorDegradesGracefully() {
    Test.setMock(HttpCalloutMock.class, new FulfillmentMock500());
    Test.startTest();
    String result = OrderStatusService.getStatus('4521');
    Test.stopTest();
    System.assertEquals('UNAVAILABLE', result,
        'A 5xx must degrade, never throw to the user');
    // and assert the log record exists: failures must leave evidence
}

Notice what the second test protects. Not data. Behavior under stress: the user's experience when the far side is down, and the audit trail that lets someone debug it tomorrow morning. Those are the two things production actually asks of an integration.

The checklist I hold callout code to

  1. A mock for 200, a mock for 5xx, and a mock for the ugliest realistic 4xx. Three behaviors, three mocks.
  2. Every test asserts an outcome. Coverage without assertions is a rounding error wearing a green checkmark.
  3. An explicit timeout on every request. The default is not a decision.
  4. Failures write a log record with a correlation Id, so one request can be traced across both systems. The test asserts the log exists.
  5. If the callout is async, the enqueue sits inside Test.startTest and stopTest, which forces the queued work to run so the assertions see real results.
  6. Malformed-body test: a 200 with garbage JSON. External systems break contracts more often than they return errors.

The opinion part

Teams sometimes treat failure-path tests as polish to add later. I think that ordering is backwards. The happy path announces its own failures loudly, in front of users. The failure paths fail silently, in log files nobody reads, at hours nobody is watching. Test coverage should be distributed by where silence is most dangerous, and by that measure the 500 deserves your first mock, not your last.

Patterns built and verified in my CaseDesk project org. If you handle this differently, tell me on LinkedIn, integration war stories are the best kind.
