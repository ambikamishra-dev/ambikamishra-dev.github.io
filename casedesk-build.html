Building CaseDesk: field notes from standing up a Service Cloud org

Ambika Mishra · 2026 · running log, updated as layers ship · code on GitHub ↗

CaseDesk is a support hub for a fictional consumer-goods manufacturer, and it exists because tutorials lie by omission. They show the happy path of each feature in isolation. Real orgs are the connections between features, and the traps live in the connections. This is the running log of building one end to end: security, automation, an Apex trigger framework, an external integration, and an Agentforce agent, with every trap documented as it fired.

Layer 1: security that survives real logins

The data model is small on purpose: Case with two record types (Product Question, Order Issue) and a custom Order__c with a lookup to Account. Lookup, not master-detail, because orders can exist without an account and must survive account deletion. The accepted cost is losing roll-up summary fields, which becomes an aggregate-query job later. Naming the tradeoff is the design work; the clicks are just clicks.

Security is Private OWD on both objects, a Support Manager and Support Agent role pair, one permission set for object access, and exactly one sharing rule. The rule does not point at the manager. The role hierarchy already grants managers visibility into subordinates' records for free, so a rule aimed at the manager would be dead configuration. The rule points sideways instead, at a sibling support branch the hierarchy can never reach.

Then the step that makes it real: log in as each test user. The agent saw only her cases. The manager saw the agent's cases with zero rules naming her. The agent could not see the manager's cases, because the hierarchy grant is a one-way street. Ten minutes of logins settles what an hour of documentation leaves fuzzy. The full mental model is in its own note.

Trap: user licenses gate everything. A Force.com license cannot touch the Case object no matter what permissions you stack on it, and an agent-type license cannot be a human user. License first, then profile, then permissions. It is the outermost gate of the security model and the one nobody mentions.

Layer 2: the escalation flow, and where configuration should live

When a case's priority becomes High, a record-triggered flow reassigns it to an Escalations queue and notifies the support manager. Two design points carry the weight. First, after-save, because the flow acts beyond the record itself; before-save can only modify the triggering record. Second, no hardcoded Ids anywhere. The queue and the custom notification type are both resolved at runtime by DeveloperName through Get Records elements. Record Ids differ between sandbox and production; API names travel. A hardcoded Id is a deployment failure with a delay timer on it.

Trap: Case record types require a Support Process to exist first, a Case-only rule (Opportunities want a Sales Process, Leads a Lead Process). And record types cannot be deleted while any profile holds them as a default: unhook the profile defaults, deactivate, then delete. Deletion has an order of operations.

Layer 3: grounding an Agentforce agent, the parts the docs skim

The agent runs on the current builder generation: subagents instead of the old topics, each with its own instructions and actions, all running as a dedicated agent user whose permissions are managed like any other user. The Q&A subagent answers from five published Knowledge articles through the standard Answer Questions with Knowledge action.

Publishing articles is not grounding. The agent reads from an Agentforce Data Library, and the library has opinions:

  • A Knowledge field can be an identifying field (how search finds the right article) or a content field (what the agent actually reads), never both. Put the body field in content, or the agent retrieves articles and finds nothing inside them.
  • The data-category filter only surfaces articles actually tagged with the selected categories. Untagged articles silently vanish from retrieval.
  • Indexing is a pipeline (data lake, model mapping, search index, retriever) and it is not instant. Testing before the pipeline shows green produces convincing false negatives.

Instructions are the other half. The template ships with sensible reasoning lines; the customization is additive: answer only from Knowledge, keep it under four sentences, and, since this company runs no live channel, replace the template's live-agent escalation with case creation. Watching the trace afterwards was the best part: on one question the first draft evaluated as ungrounded, the agent re-reasoned, and the second draft passed as grounded with a cited source. The output evaluation loop is not marketing; it is visible in the trace panel.

What ships next

The trigger framework on Case (one trigger per object, handler and service layers, recursion control, a bypass permission for data loads), a nightly SLA batch, an LWC case cockpit, and the agent's hands: a Flow action that creates properly-typed cases and an invocable Apex action calling an external fulfillment API through a Named Credential, with failure paths that degrade instead of exploding. Each layer gets its entry here as it ships.
